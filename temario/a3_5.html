Temario NM · A3.5

A3.5 Administración, diseño y transacciones en bases de datos

Pregunta guía

¿Cómo se organiza, diseña, administra y protege una base de datos para que funcione de manera correcta, eficiente y segura en un sistema informático real?

Introducción

Cuando hablamos de bases de datos, muchas veces pensamos solo en tablas, registros, campos o consultas SQL. Sin embargo, una base de datos real implica mucho más que almacenar información. Detrás de cualquier sistema bien construido hay personas, procesos y decisiones técnicas que permiten que los datos estén organizados, protegidos y disponibles cuando se necesitan.

Por ejemplo, un hospital, una tienda online o una plataforma educativa no solo necesitan guardar datos. También necesitan:

  • diseñar correctamente la estructura de la base de datos
  • decidir cómo se relacionan los datos
  • mantener la seguridad de la información
  • asegurar que varios usuarios puedan trabajar al mismo tiempo sin provocar errores
  • proteger la base de datos frente a fallos o pérdidas de información

En este contexto aparecen varias ideas fundamentales:

  • el administrador de bases de datos
  • el diseño conceptual
  • el diseño lógico
  • el diseño físico
  • las transacciones
  • las propiedades ACID

Todos estos conceptos forman parte de la gestión real de las bases de datos y son esenciales para comprender cómo funcionan los sistemas informáticos en el mundo profesional.

Explicación detallada

1. El administrador de bases de datos

¿Qué es un administrador de bases de datos?

El administrador de bases de datos, conocido también como DBA por sus siglas en inglés (Database Administrator), es la persona o perfil técnico responsable de gestionar, mantener, proteger y supervisar una base de datos.

No se encarga solo de “guardar datos”, sino de asegurar que la base de datos funcione correctamente, sea segura y esté disponible cuando los usuarios o aplicaciones la necesiten.

En sistemas pequeños, una misma persona puede realizar varias funciones al mismo tiempo. En sistemas grandes, el rol del administrador de bases de datos suele ser muy importante y especializado.

Idea clave

El DBA no diseña únicamente tablas. Su trabajo consiste en garantizar que la base de datos sea:

  • estable
  • segura
  • eficiente
  • recuperable en caso de fallo
  • accesible para quienes deben usarla
Ejemplo real

Imagina la base de datos de un hospital. En ella se almacenan:

  • pacientes
  • citas
  • historiales médicos
  • tratamientos
  • resultados de pruebas

Si la base de datos falla, los médicos podrían perder acceso a información crítica. Si no está bien protegida, podrían filtrarse datos sensibles. Si no está bien mantenida, el sistema podría volverse lento o inestable. Ahí es donde entra la figura del administrador de bases de datos.

Funciones del administrador de bases de datos

El trabajo de un DBA puede incluir muchas tareas. Las más importantes son estas:

1. Instalación y configuración del sistema gestor de bases de datos

El DBA se encarga de instalar y configurar el sistema gestor de bases de datos, como por ejemplo:

  • MySQL
  • PostgreSQL
  • Oracle
  • SQL Server

Esto implica dejar preparado el entorno para que la base de datos pueda funcionar correctamente.

2. Gestión de usuarios y permisos

No todas las personas deben poder hacer lo mismo dentro de una base de datos.

Por ejemplo:

  • un profesor puede consultar notas
  • un alumno quizá solo puede ver sus propias calificaciones
  • el personal administrativo puede modificar matrículas
  • el administrador tiene acceso completo

El DBA define qué usuarios existen y qué permisos tiene cada uno.

3. Seguridad de la base de datos

Una base de datos debe protegerse frente a:

  • accesos no autorizados
  • modificaciones indebidas
  • pérdida de información
  • ataques externos

El administrador debe aplicar medidas de seguridad para proteger los datos.

4. Copias de seguridad y recuperación

Una de las funciones más importantes del DBA es realizar copias de seguridad o backups.

Si se produce un fallo del servidor, un error humano o un ataque, estas copias permiten recuperar la información.

5. Supervisión del rendimiento

Si una base de datos funciona lentamente, toda la aplicación puede verse afectada.

El administrador analiza el rendimiento y busca mejoras para que las consultas, inserciones y actualizaciones se realicen de forma eficiente.

6. Mantenimiento general

Esto puede incluir:

  • actualizar el sistema
  • revisar errores
  • optimizar estructuras
  • supervisar el uso del almacenamiento
  • comprobar la integridad de los datos

7. Apoyo al diseño y evolución de la base de datos

En muchos casos, el DBA también participa en la mejora del diseño de la base de datos a medida que el sistema crece o cambia.

Ejemplo de funciones del DBA en una plataforma educativa

En una plataforma escolar, el administrador de bases de datos podría encargarse de:

  • crear la base de datos del sistema
  • gestionar usuarios como profesorado, alumnado y administración
  • controlar quién puede ver o modificar las calificaciones
  • hacer copias de seguridad semanales
  • recuperar datos si se borran por error
  • asegurar que la plataforma siga funcionando durante los exámenes

2. Diseño conceptual de una base de datos

¿Qué es el diseño conceptual?

El diseño conceptual es la primera fase importante en la construcción de una base de datos.

En esta fase no se piensa todavía en tablas concretas, tipos de datos técnicos ni comandos SQL. Lo que se hace es identificar qué información necesita el sistema y cómo se relacionan los elementos principales.

Es una visión general y abstracta de la base de datos.

Objetivo del diseño conceptual

El objetivo es responder preguntas como:

  • ¿Qué entidades existen en el sistema?
  • ¿Qué información importante debe guardarse?
  • ¿Cómo se relacionan esas entidades?
Entidades

Una entidad es un objeto, elemento o concepto sobre el que queremos almacenar información.

Ejemplos de entidades:

  • Alumno
  • Profesor
  • Curso
  • Asignatura
  • Pedido
  • Cliente
  • Producto
Relaciones

Las entidades suelen estar conectadas entre sí.

Por ejemplo:

  • un alumno se matricula en un curso
  • un profesor imparte una asignatura
  • un cliente realiza pedidos

Estas conexiones se representan mediante relaciones.

Ejemplo conceptual: biblioteca

Supongamos que queremos diseñar una base de datos para una biblioteca.

Podríamos identificar estas entidades:

  • Libro
  • Usuario
  • Préstamo

Y estas relaciones:

  • un usuario puede realizar préstamos
  • un préstamo corresponde a un libro
  • un préstamo está asociado a un usuario
Herramienta habitual: diagrama entidad-relación

El diseño conceptual suele representarse mediante un diagrama entidad-relación.

Este diagrama muestra:

  • entidades
  • atributos principales
  • relaciones entre entidades

No entra aún en detalles técnicos del sistema gestor.

Importancia del diseño conceptual

El diseño conceptual es importante porque permite entender el sistema antes de construirlo.

Si esta fase se hace mal, después pueden aparecer problemas como:

  • falta de información relevante
  • relaciones incorrectas
  • diseño poco claro
  • dificultades para ampliar el sistema

3. Diseño lógico de una base de datos

¿Qué es el diseño lógico?

El diseño lógico transforma el diseño conceptual en una estructura más cercana a la implementación real.

Aquí ya se empieza a pensar en:

  • tablas
  • campos
  • claves primarias
  • claves externas
  • relaciones entre tablas

Sin embargo, todavía no se entra en detalles físicos de almacenamiento.

Objetivo del diseño lógico

El objetivo es convertir las ideas generales del diseño conceptual en una estructura organizada que pueda implementarse en un sistema relacional.

Ejemplo

Siguiendo con la biblioteca:

Diseño conceptual

Entidades:

  • Libro
  • Usuario
  • Préstamo

Diseño lógico

Tablas:

  • Libros
  • Usuarios
  • Prestamos

Tabla Libros

  • ID_libro
  • titulo
  • autor

Tabla Usuarios

  • ID_usuario
  • nombre
  • email

Tabla Prestamos

  • ID_prestamo
  • ID_usuario
  • ID_libro
  • fecha_prestamo
  • fecha_devolucion

Aquí ya aparecen claramente:

  • claves primarias
  • claves externas
  • relaciones
Relación con la normalización

En el diseño lógico también es muy importante revisar la normalización para evitar:

  • redundancias
  • inconsistencias
  • problemas de actualización
Importancia del diseño lógico

El diseño lógico sirve para construir una base de datos coherente y bien estructurada antes de implementarla físicamente.

Si esta etapa está bien hecha, será mucho más fácil:

  • crear las tablas
  • hacer consultas
  • mantener la integridad de los datos
  • ampliar la base de datos en el futuro

4. Diseño físico de una base de datos

¿Qué es el diseño físico?

El diseño físico es la fase en la que la base de datos se concreta técnicamente dentro de un sistema gestor real.

Aquí ya no solo pensamos en qué tablas habrá, sino en cómo se almacenarán físicamente los datos.

Aspectos del diseño físico

El diseño físico incluye decisiones como:

  • tipo exacto de datos de cada campo
  • tamaño de los campos
  • índices
  • organización del almacenamiento
  • particionado
  • rendimiento
  • acceso físico a los datos
Ejemplo

En el diseño lógico habíamos decidido que existe un campo nombre en la tabla Usuarios.

En el diseño físico ya tendríamos que decidir cosas como:

  • si será VARCHAR(100)
  • si puede ser nulo o no
  • si debe tener índice
  • cómo se almacenará en disco
Índices

Un índice es una estructura que acelera las búsquedas dentro de la base de datos.

Por ejemplo, si buscamos constantemente usuarios por email, podría ser útil crear un índice sobre ese campo.

Eso mejora el rendimiento, aunque también puede aumentar el coste de ciertas operaciones de escritura.

Importancia del diseño físico

El diseño físico influye mucho en:

  • velocidad de las consultas
  • uso del espacio
  • eficiencia general del sistema
  • escalabilidad de la base de datos

Un mal diseño físico puede hacer que una base de datos funcione lentamente aunque su diseño conceptual y lógico sean correctos.

5. Diferencia entre diseño conceptual, lógico y físico

Es muy importante que el alumnado entienda bien la diferencia entre estas tres fases.

Diseño conceptual

Se centra en qué información existe y cómo se relaciona, sin entrar en detalles técnicos.

Diseño lógico

Se centra en cómo organizar esa información en tablas y relaciones.

Diseño físico

Se centra en cómo implementar realmente esa estructura en el sistema gestor, teniendo en cuenta almacenamiento y rendimiento.

Resumen sencillo
  • Conceptual = idea general del sistema
  • Lógico = estructura de tablas y relaciones
  • Físico = implementación técnica real

6. Las transacciones

¿Qué es una transacción?

Una transacción es un conjunto de operaciones sobre la base de datos que deben ejecutarse como una única unidad lógica.

Esto significa que todas esas operaciones deben completarse correctamente o, si algo falla, no debe aplicarse ninguna.

Idea clave

Una transacción asegura que la base de datos no quede en un estado incorrecto o incompleto.

Ejemplo clásico: transferencia bancaria

Imagina una transferencia de 100 euros desde la cuenta A a la cuenta B.

La operación implica al menos dos acciones:

  1. restar 100 euros de la cuenta A
  2. sumar 100 euros a la cuenta B

Estas dos acciones deben formar parte de una misma transacción.

¿Qué pasaría si falla la segunda?

Si se resta el dinero de A pero no se suma a B, la base de datos quedaría en un estado incorrecto.

Por eso ambas operaciones deben tratarse como una sola unidad.

Ejemplo cotidiano

Supongamos una tienda online donde una compra implica:

  1. registrar el pedido
  2. descontar stock
  3. registrar el pago

Si solo se registra el pedido pero no se actualiza el stock, o si el pago falla y aun así se crea el pedido, el sistema quedaría inconsistente.

Una transacción evita este problema.

Inicio, confirmación y reversión

En muchos sistemas, una transacción suele seguir esta lógica:

  • BEGIN o inicio de transacción
  • ejecución de operaciones
  • COMMIT si todo sale bien
  • ROLLBACK si algo falla

COMMIT

Confirma definitivamente los cambios.

ROLLBACK

Deshace los cambios realizados durante la transacción.

7. Propiedades ACID de las transacciones

Las transacciones en bases de datos suelen cumplir las llamadas propiedades ACID. Estas propiedades garantizan que las operaciones se realicen de forma segura y fiable.

ACID significa:

  • Atomicity (atomicidad)
  • Consistency (consistencia)
  • Isolation (aislamiento)
  • Durability (durabilidad)
7.1 Atomicidad

La atomicidad significa que una transacción se ejecuta completamente o no se ejecuta en absoluto.

No puede quedarse “a medias”.

Ejemplo

En una transferencia bancaria:

  • o se resta dinero de una cuenta y se suma a la otra
  • o no se hace ninguna de las dos cosas

No se permite un estado intermedio incorrecto.

Idea clave

Todo o nada.

7.2 Consistencia

La consistencia significa que la transacción lleva la base de datos de un estado válido a otro estado válido.

Es decir, después de la transacción, las reglas del sistema deben seguir cumpliéndose.

Ejemplo

Si en una base de datos bancaria no se permiten saldos imposibles o cuentas inexistentes, una transacción no debe romper esas reglas.

Otro ejemplo

Si un préstamo debe asociarse siempre a un usuario y a un libro válidos, una transacción no debe crear un préstamo con referencias incorrectas.

Idea clave

La base de datos debe seguir siendo coherente antes y después de la transacción.

7.3 Aislamiento

El aislamiento significa que varias transacciones que ocurren al mismo tiempo no deben interferirse de forma incorrecta.

Cada transacción debe comportarse como si fuera la única que se está ejecutando, al menos desde el punto de vista lógico.

Ejemplo

Dos personas compran el último producto disponible casi al mismo tiempo.

Si la base de datos no maneja bien el aislamiento, ambas operaciones podrían pensar que el producto sigue disponible.

Idea clave

Las transacciones concurrentes no deben provocar resultados erróneos.

7.4 Durabilidad

La durabilidad significa que, una vez que una transacción ha sido confirmada con éxito, sus cambios deben permanecer guardados incluso si ocurre un fallo posterior.

Ejemplo

Si un pago ya ha sido confirmado y después se apaga el servidor, la información de ese pago no debe perderse.

Idea clave

Lo que ya fue confirmado debe mantenerse.

8. Ejemplo completo de ACID

Supongamos una compra online con una sola unidad disponible de un producto.

La transacción incluye:

  1. comprobar stock
  2. registrar pedido
  3. descontar stock
  4. confirmar compra

Atomicidad

Si falla el descuento de stock, toda la operación se revierte.

Consistencia

Después de la compra, el stock no puede quedar en un valor inválido.

Aislamiento

Si dos personas compran a la vez, el sistema debe evitar vender dos veces la misma unidad.

Durabilidad

Una vez confirmada la compra, el pedido debe seguir registrado aunque falle el sistema.

9. Importancia de las transacciones en sistemas reales

Las transacciones son esenciales en sistemas donde los datos son críticos, por ejemplo:

  • bancos
  • hospitales
  • comercio electrónico
  • reservas de viajes
  • plataformas educativas
  • sistemas administrativos

En todos estos casos, un error parcial puede tener consecuencias graves.

10. Ejemplo integrado: universidad

Imaginemos un sistema universitario.

El DBA

Se encarga de:

  • gestionar usuarios
  • hacer backups
  • controlar accesos
  • mantener el sistema

Diseño conceptual

Se identifican entidades como:

  • Alumno
  • Asignatura
  • Matrícula
  • Profesor

Diseño lógico

Se crean tablas:

  • Alumnos
  • Asignaturas
  • Matriculas
  • Profesores

Diseño físico

Se definen:

  • tipos de campos
  • índices
  • restricciones
  • configuración del almacenamiento

Transacciones

Una matrícula puede implicar:

  1. registrar la matrícula
  2. actualizar el número de plazas disponibles
  3. generar recibo

Si algo falla, se deshace toda la operación.

Propiedades ACID

Garantizan que esa matrícula no deje el sistema en estado incorrecto.

Conceptos clave

  • DBA: administrador de bases de datos responsable de la gestión, seguridad y mantenimiento del sistema.
  • Diseño conceptual: fase en la que se identifican entidades, atributos y relaciones de forma general.
  • Diseño lógico: fase en la que se organizan tablas, claves y relaciones.
  • Diseño físico: implementación técnica real de la base de datos en el sistema gestor.
  • Transacción: conjunto de operaciones que debe ejecutarse como una sola unidad lógica.
  • COMMIT: confirmación definitiva de los cambios de una transacción.
  • ROLLBACK: reversión de los cambios cuando ocurre un fallo.
  • ACID: conjunto de propiedades que garantizan transacciones seguras y fiables.

Ejemplo práctico

Supongamos una plataforma universitaria donde un estudiante se matricula en una asignatura.

El proceso puede incluir:

  1. comprobar si quedan plazas disponibles
  2. registrar la matrícula del alumno
  3. actualizar el número de plazas restantes
  4. generar un justificante

Estas operaciones forman una transacción, porque si falla una de ellas, no debe quedar el sistema en un estado incoherente.

Además:

  • el DBA se encarga de que la base de datos esté disponible y protegida
  • el diseño conceptual identifica entidades como Alumno, Asignatura y Matrícula
  • el diseño lógico organiza tablas como Alumnos, Asignaturas y Matriculas
  • el diseño físico concreta tipos de campos, índices y almacenamiento
  • las propiedades ACID garantizan que la matrícula se procese correctamente

Aplicaciones en el mundo real

  • Bancos: transferencias, pagos y movimientos de cuentas dependen de transacciones seguras.
  • Hospitales: historiales clínicos, citas y tratamientos requieren acceso controlado y consistencia.
  • Comercio electrónico: pedidos, stock y pagos deben coordinarse sin errores.
  • Plataformas educativas: matrículas, calificaciones y usuarios necesitan una gestión fiable.
  • Sistemas administrativos: expedientes, permisos y registros requieren seguridad y trazabilidad.

En todos estos casos, el buen diseño y la administración correcta de la base de datos resultan esenciales.

Errores frecuentes de los estudiantes

  • Confundir DBA con programador: el administrador de bases de datos no se limita a programar consultas; gestiona seguridad, copias, rendimiento y mantenimiento.
  • Confundir diseño conceptual con diseño lógico: el conceptual identifica entidades y relaciones; el lógico ya organiza tablas y claves.
  • Pensar que el diseño físico es solo “guardar archivos”: en realidad incluye decisiones técnicas que afectan al rendimiento y al almacenamiento real.
  • Creer que una transacción es una sola consulta: una transacción puede incluir varias operaciones que deben ejecutarse juntas.
  • Memorizar ACID sin entenderlo: lo importante no es solo recordar las siglas, sino comprender qué problema resuelve cada propiedad.

  • A3.3 Programación de bases de datos: las transacciones y el trabajo real con bases de datos se relacionan con las consultas SQL y la manipulación de datos.
  • A3.4 Normalización: el diseño lógico se apoya mucho en una buena normalización para evitar redundancias e inconsistencias.

Resumen

  • El administrador de bases de datos (DBA) se encarga de gestionar, proteger y mantener la base de datos.
  • Sus funciones incluyen seguridad, copias de seguridad, permisos, rendimiento y mantenimiento.
  • El diseño conceptual identifica entidades y relaciones.
  • El diseño lógico transforma esa visión en tablas, claves y relaciones estructuradas.
  • El diseño físico concreta la implementación técnica real en el sistema gestor.
  • Una transacción es un conjunto de operaciones que deben ejecutarse como una sola unidad.
  • Las propiedades ACID garantizan fiabilidad:
  • Atomicidad: todo o nada
  • Consistencia: la base de datos sigue siendo válida
  • Aislamiento: las transacciones no interfieren incorrectamente entre sí
  • Durabilidad: los cambios confirmados permanecen
